## 1. Действительно ли код ускорен этой волной?

### Короткий ответ

**Да, но узко: для одного класса нагрузок и с одной существенной
методологической оговоркой, которую gate-доки не раскрывают.**
В `production` этой волной вошло ровно одно перф-изменение —
`class-aware-dirty` (R13-9, `da77b38`). Всё остальное — либо
correctness-фиксы без измеримого ускорения (R13-1/2/3/11/12), либо opt-in
фичи, которые сознательно НЕ промоутнуты (R13-6 пара — из-за реальной
регрессии; R13-7 — без production-гейта вовсе).

### 1.1 `class-aware-dirty` в production — числа подтверждаются, headline требует оговорки

Заявленные числа сверены с raw-логами
(`_raw_r13_9_wallclock_baseline_off.log` / `_raw_r13_9_wallclock_treatment_on.log`)
— напечатанные харнессом значения совпадают с процитированными в
`docs/perf/R13_9_CLASS_AWARE_DIRTY_PRODUCTION_GATE.md` §3 байт-в-байт:

| N producers | baseline ns/owner_alloc | treatment | заявленный выигрыш |
|---:|---:|---:|---:|
| 1 | 722.7 | 722.1 | 1.00× |
| 2 | 909.2 | 835.4 | 1.09× |
| 4 | 1 367.3 | 979.2 | 1.40× |
| 8 | 23 527.4 | 1 083.9 | **21.71×** |

iai-числа тоже сверены (`_raw_r13_9_iai_*.log`): +0.00…0.02% Ir на всех 12
не-remote бенчах, фиксированные +5 Ir независимо от размера бенча —
корректно интерпретировано как «фича вкомпилирована, но путь не достигается»,
т.е. промоушен действительно бесплатен для однопоточных горячих путей.

**Найденная оговорка (P1, методология измерения).** Метрика
`ns/owner_alloc` считается харнессом
(`benches/r12_7_class_aware_dirty_wallclock.rs:281-315`) по **ручному
таймингу под-окна** внутри раунда (`start = Instant::now()` на строке 181 —
ПОСЛЕ pre-аллокации 6400 блоков; `recycle` — после остановки таймера).
Criterion же меряет полный раунд. Из тех же raw-логов:

| N=8 | полный раунд (criterion, mean) | тайм-окно харнесса (ns/round) |
|---|---:|---:|
| baseline | ~20.6 ms | 18.8 ms |
| treatment | ~18.4 ms | **1.35 ms** |

То есть end-to-end время раунда улучшилось лишь на ~11% (20.6→18.4 ms), а
не в 21.7×: в treatment ~17 ms работы (drain отложенных remote-free)
**переехало в неизмеряемую часть раунда** — pre-alloc следующей итерации и
`HeapRegistry::recycle`. При N=4 полный раунд вообще почти не изменился
(1.840 → 1.811 ms, ~1.6%), при том что метрика окна показывает 1.40×.

Это НЕ отменяет промоушен: выигрыш в latency аллокаций владельца в окне
конкуренции реален (владелец перестаёт впустую сканировать O(D) чужих
dirty-сегментов), стоимость вне remote-пути ~нулевая, RSS-цена 8 KiB/heap
корректно перемерена (включая честную поправку 6.1→8.0 KiB к R12-7). Но
формулировка «~20-32× faster on cross-thread remote-free-heavy workloads»
(CHANGELOG, R13_WAVE_SUMMARY §6, комментарий в Cargo.toml) описывает
метрику под-окна, а не сквозную пропускную способность; ни один из
документов (R12-7, R13-9, wave summary) не приводит criterion-время полного
раунда рядом с headline. Рекомендация — в §3.1 ниже.

### 1.2 Остальное — честно НЕ ускорение, и это прямо задокументировано

- **R13-3 (virgin-zero-skip через magazine):** собственный gate
  (`docs/perf/R13_3_VIRGIN_ZERO_SKIP_MAGAZINE_GATE.md`) прямо говорит: «No
  scenario shows a statistically significant difference» (p=0.39…0.73).
  Ценность коммита — correctness (Defect 2: heap, зовущий только
  `alloc_zeroed`, никогда не дренировал `HeapOverflow` — ресурсная
  ретенция) и структурная унификация пути. Заявлено честно, не продано как
  ускорение. Фича остаётся opt-in.
- **R13-6 (exact-span-large + large-reserved-capacity):** НЕ промоутнуто —
  и правильно: сверено по `_raw_r13_6_iai_*.log`, `realloc_grow`
  513 187 → 1 038 281 Ir = **+102.3%** (детерминированно). RSS-выигрыш
  (15.80×→1.06× на 260 KiB) реален, но заблокирован регрессией. Вердикт
  CONDITIONAL-GO обоснован.
- **R13-7 (large-cache-extended):** «~99×» (и «278×» в R13-8 §4) — только
  на opt-in фиче и только на узком turnover-профиле (циклы alloc-all/
  dealloc-all по 9-24 различным Large-размерам); R13-8 отдельно подтвердил
  0 хитов на статическом live-set. Production не затронут.
- **R13-2 (NUMA bucket reuse):** correctness/ёмкость (`numa-aware` вообще
  не в production), не скорость.

### 1.3 README-таблица: абсолютные числа несравнимы между запусками

Рефреш таблицы (R13-10) — правильный процесс, но дифф README показывает,
что абсолютные ns выросли в ~1.6× у **всех трёх** аллокаторов между
2026-07-20 и 2026-07-23 (churn+write 16 B: Sefer 14.7→24.2, mimalloc
13.9→22.7, System 83.6→144.4 ns) — это состояние хоста, не код. Соотношения
(колонка «vs mimalloc») остаются валидным сигналом; абсолютные колонки —
нет. Wave summary §6 честно называет отдельные свинги «not attributed to
any Round13 code change», но сама таблица в README подаёт абсолюты как
основной контент. См. §4.2.

### Итоговый вердикт по вопросу 1

Волна дала: (а) один реальный production-выигрыш — многократное снижение
latency аллокаций владельца под cross-thread remote-free нагрузкой при
N≥4 конкурирующих классах-продюсерах, с нулевой ценой для всего остального
(лучшее из подтверждённого: 1.40× на N=4 и 21.7× на N=8 по метрике окна;
сквозной wall-clock раунда на харнессе — единицы процентов); (б) пять
P0/P1 correctness-фиксов в уже шипящемся коде; (в) два измеренных, но
сознательно не промоутнутых opt-in ускорения. Однопоточные горячие пути не
изменились вовсе — и это подтверждено детерминированным iai, а не только
wall-clock.

## 2. Что ещё можно сильно ускорить (направления, по убыванию потенциала)

**S1 (P0 по потенциалу). Разблокировать exact-span-large для production —
адаптивный `LARGE_RESERVED_CAP_GROWTH_FACTOR`.** Единственный блокер пары —
+102.3% Ir на doubling-realloc из-за фиксированного 2× потолка
`reserved_capacity`, ре-трипающегося почти на каждом шаге удвоения.
R13-6 §7 уже даёт готовый план: расширить до 4× или компаундить множитель
через релокации; оракул (`perf_gate_iai::realloc_grow`) уже существует.
Приз: RSS-амплификация 15.80×→1.06× на 260 KiB Large в production —
крупнейший недобранный выигрыш проекта прямо сейчас.

**S2 (P1). Cold-direct 16 B / 256 B — 2.4-2.7× позади mimalloc**
(собственная таблица README, стабильно через все раунды). Page-run слой
переведён в DEFERRED «нет production-жертвы» (R13-4), но R13-8 как раз
предъявил жертву — см. S3; связка «page-run или расширяемая таблица»
заслуживает переоценки как единое решение и для cold-carve, и для потолка.

**S3 (P1). Жёсткая стена `MAX_SEGMENTS`: ровно 1023 живых Large-объекта**
— 100% воспроизводимо во всех фичевых конфигурациях (R13-8 §0/§3;
1024-й слот съеден primordial-сегментом). Для процессов с сотнями-тысячами
одновременно живых 260 KiB-2 MiB объектов это не деградация, а отказ
(`alloc()` → null). Расширяемая `SegmentTable` (или page-run) — не только
перф, но и capacity-корректность.

**S4 (P2). Сквозная пропускная способность drain-пути.** Раз full-round на
r12_7-харнессе почти не изменился (§1.1), следующий уровень выигрыша — не
маршрутизация сканов, а сам объём drain-работы: батчинг реклейма из
ring/overflow, амортизация `sync_directory_for_segment_classes` по
сегменту вместо по-блочной. Сначала — инструментировать, куда уходят ~17 ms
неизмеряемой части раунда treatment-арма.

**S5 (P2). Одноразовый `sidecar_oom_latch` (R13-1) — heap, один раз
поймавший sidecar-OOM, навсегда теряет весь per-class выигрыш** (coarse-only
до конца жизни heap'а). Для correctness это правильная минимальная починка,
но recovery-путь (пере-материализация с ре-синком) вернул бы 21× тем
heap'ам, которым не повезло в момент memory pressure.

**S6 (P2). Production-гейт для `large-cache-extended`.** 99×/278× на
turnover-профиле измерены, вред нигде не показан, но A/B-гейта в стиле
R13-6/R13-9 не было (R13-7 сам это фиксирует). Дешёвая задача: механика
гейта уже отработана.

**S7 (P3). 16 B churn+write — 1.07× позади mimalloc** — последний
проигрышный размер на паттерне переиспользования; после S1/S2 стоит
посмотреть на цену магазинного pop для минимального класса.

## 3. Что улучшить в коде (приоритизировано)

**P1 — soundness: zeroed-байты как `[Option<CachedLarge>; 32]`
(`src/alloc_core/large_cache_extended.rs:116-143` +
`src/alloc_core/alloc_core_large_cache.rs`, `large_cache_find_free_slot`).**
Сайдкар резервируется `leak_zeroed_pages` и сразу читается через
`deref_large_cache_extension(...).slots[i].as_ref()` / `.iter().position(|s|
s.is_none())` — т.е. код **опирается** на то, что all-zero байты — валидный
`None` для `Option<CachedLarge>`. Но `CachedLarge`
(`src/alloc_core/alloc_core.rs:185-213`) состоит из голых `*mut u8`/`usize`
— ниши нет, а layout дискриминанта `repr(Rust)`-enum без ниши не
специфицирован. Сегодняшний rustc так делает, гарантии — нет. Доc-коммент
(`large_cache_extended.rs:130-137`) сам противоречив: заявляет «this is NOT
relied upon», хотя код именно на это полагается. У соседних сайдкаров
(`SegmentDirectory`, `PerClassDirty`) проблемы нет — там только
целые/атомики, для которых zeroed валиден. **Фикс дешёвый:** сразу после
`reserve_large_cache_extension()` явно записать
`ptr::write(p, LargeCacheExtension { slots: [const { None }; N] })`
(~1.8 KiB однократно), либо перевести поля `CachedLarge` на
`NonNull`+`Option`. Плюс переписать вводящий в заблуждение коммент.

**P2 — API-футган: safe fn → `&'static mut` из `*mut`
(`large_cache_extended.rs:152-183`, унаследовано от
`os::deref_directory_sidecar[_mut]`, `src/alloc_core/os.rs:470-495`).**
`deref_large_cache_extension_mut` — безопасная функция с контрактом только
в док-комменте; два вызова подряд из safe-кода внутри crate дают алиасящие
`&'static mut` — UB без единого `unsafe` на сайте вызова. По собственному
tier-2 правилу проекта (контракт в сигнатуре, не в прозе — README §"Where
unsafe lives") таким функциям место быть `unsafe fn`. Это уже существующий
паттерн (техдолг), но R13-7 его размножил в третий модуль — стоит завести
задачу на конверсию всех трёх сайдкар-deref-пар в `unsafe fn`.

**P2 — харнесс r12_7/r13_9: печатать сквозные числа рядом с метрикой окна
(`benches/r12_7_class_aware_dirty_wallclock.rs:281-315`).** См. §1.1:
ручной `ns/owner_alloc` и criterion-время полного раунда расходятся до
13× в treatment-арме, и только первый попадает в headline. Добавить в
вывод харнесса «full-round (criterion-comparable) ms» и обязать gate-доки
цитировать обе оси; ретроактивно — одну поясняющую сноску в
`R13_9_CLASS_AWARE_DIRTY_PRODUCTION_GATE.md` §3 и `R13_WAVE_SUMMARY.md` §2.

**P3 — мелочи (не блокеры):**
- `alloc_core_small_magazine.rs` (R13-3): `((1u32 << n) - 1) as u16` —
  корректно при `TCACHE_CAP = 16` (`src/registry/tcache.rs:48`), но связь
  «маска u16 ⟺ CAP ≤ 16» держится только на соглашении; дешёвый
  `const _: () = assert!(TCACHE_CAP <= 16)` рядом с `virgin_mask` зафиксировал
  бы её на компиляции.
- `R13_WAVE_SUMMARY.md` §2 называет формат «A/B/B/A», но фактически это
  A/B с двойной перепроверкой цитат (сам документ это честно оговаривает) —
  название слегка переобещает протокол.

**Что в коде волны сделано хорошо (заслуживает фиксации как норма):**
R13-12 — образцовое минимальное гейтирование, зеркалящее все сиблинг-сайты;
R13-2 — счётчик по «предыдущему значению бита», а не по заявлению
вызывающего, плюс red/green counterfactual-тесты; R13-3 — структурное
слияние путей вместо параллельной ветки; R13-7 — три самопойманных дефекта
методологии измерения, честно описанных в коммите.

## 4. Что улучшить в проекте (процессы, тесты, CI, документация)

**4.1 (P1) CI: структурная защита от R13-12-класса ошибок.** E0599 на
комбинации `alloc-xthread+fastbin+alloc-decommit` без
`alloc-segment-directory` прожил незамеченным до случайного обнаружения.
R13-5 добавил конкретные feature-isolation строки — это лечит известные
комбинации. Системное решение — `cargo hack check --feature-powerset`
(хотя бы с `--depth 2`) как отдельный CI-job на `cargo check`-уровне:
дёшево и ловит весь класс «метод существует не во всех комбинациях».

**4.2 (P1) Шум канонической wall-clock таблицы.** ±60% дрейфа абсолютных
ns между двумя запусками (см. §1.3) обесценивает абсолютные колонки
README. Варианты: (а) пометить в шапке таблицы, что нормативны только
соотношения, абсолюты — иллюстрация одного запуска; (б) добавить в
`scripts/bench-table.mjs` «якорь калибровки хоста» (фиксированный
эталонный цикл, печатаемый рядом с таблицей) — расхождение якоря между
запусками сразу маркирует несравнимость; (в) по возможности фиксировать
power plan/частоту на время прогона. iai уже правильно назначен
детерминированным судьёй — довести это до статуса «абсолюты только из iai».

**4.3 (P2) Правило для перф-гейтов: две оси всегда.** В CLAUDE.md уже есть
раздел про перф-волны; добавить пункт: «wall-clock gate обязан приводить и
метрику под-окна, и сквозное criterion-время того же харнесса; расхождение
между ними — само по себе результат, подлежащий объяснению». R13-9 —
живой пример, где это правило изменило бы формулировку headline.

**4.4 (P2) Вес закоммиченных raw-логов.** Политика R13-10 (.gitignore +
`git add -f` только процитированного) — правильная. Но по ней в этой волне
закоммичено ~35 файлов, включая два полных bench-table лога по ~1 935 строк
(`_raw_r13_10_bench_table_full.log`,
`_raw_r13_6_bench_table_production_baseline.log`). Стоит дополнить политику:
цитируемый лог можно усекать до релевантных секций с пометкой об усечении —
иначе docs/perf продолжит расти на мегабайты за волну.

**4.5 (P3) Гигиена волны.** (а) Диапазон в отчётах волны лучше фиксировать
после финального пуша: `cb24266` уже недостижим с ветки (заменён
`c7e7a79`), и цитата диапазона в постановке задач устаревает. (б) В рабочем
дереве на момент ревью — незакоммиченный `docs/reviews/2026-07-23-r14-plan.md`;
не потерять при следующем коммите.

**Что в процессе волны сделано отлично:** wave summary как единая точка
ответа «что изменилось в production»; честные CONDITIONAL-GO/не-промоушены
с конкретными условиями снятия; исправление собственного прошлого числа
(6.1→8.0 KiB) вместо замалчивания; понижение «SUPERSEDED»→«DEFERRED» по
внешнему ревью; counterfactual-верификация тестов как норма; фиксация
самопойманных ложных тревог (§1.5 R13-9) вместо тихой правки.
