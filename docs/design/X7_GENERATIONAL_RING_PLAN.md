# X7 — hardened generational ring entries: план реализации (Ф0–Ф5)

**Статус:** план утверждён (2026-07-06), реализация не начата. Задача #188.
**Предшественник:** `RING_MAGAZINE_XTHREAD_DOUBLE_FREE_FIX.md` §8 (теорема о
неразличимости + costed-набросок §8.4). Этот файл — детализация до фаз.

## 1. Что закрываем — и что НЕ закрываем (главный инсайт)

Теорема §8 остаётся в силе: remote free стейл-указателя, **вызванный после**
переиздания блока, информационно неотличим от легального remote free текущей
жизни — на голом `GlobalAlloc`-интерфейсе это граница навсегда (поколение
негде передать в указателе; типизированное решение — Region-face).

> **Поправка (R1, 2026-07-06):** таксономия теперь ТРИ ноги, не две. X2
> (#164) закрыл ногу 1 (in-magazine-at-drain). R1 (этот таск, ретро C1)
> закрыл ногу 2 — refill-window in-out-buffer: предикат X2 был слеп к блокам,
> уже вытянутым в `out[0..filled]` в ходе ТЕКУЩЕГО refill (предикат говорит
> `if k == c { return false; }` — аргумент borrow-безопасности, не
> семантическое покрытие). Фикс R1 — оборачивание предиката guard-ом
> `is_in_magazine(ptr,k) || (k == c && out[..filled].contains(ptr))` —
> дёшев, без поколений, ноль на пустом ринге. Нога 2 ИНФОРМАЦИОННО
> РАЗЛИЧИМА (блок виден в out-буфере), потому теорема §8 её НЕ покрывает.
> Осталась только нога 3 — мгновенная гонка вызова (ниже).

X7 нацелен ТОЛЬКО на ногу 3 (re-issue-before-drain / мгновенная гонка):

```
сейчас:  окно гонки = [записка легла в ринг] ──────────── [ленивый дренаж]
                        блок успел переиздаться где-то ЗДЕСЬ → коррупция
с X7:    окно гонки = [сам вызов remote free] ↔ [переиздание]
```

**Ленивость дренажа перестаёт быть усилителем гонки.** Записка, «пережившая»
переиздание в ринге (интерливинг A→B→I→D), опознаётся по несовпадению
поколений и выбрасывается. Остаётся только мгновенная гонка самого вызова
(B→I→A→D) — теорема, честно документируется.

**Критерий успеха, объявленный заранее:** старый красный `#[ignore]`-тест
`residual_xthread_double_free_no_corruption` (сценарий A→B→I→D) под
`hardened` СТАНОВИТСЯ ЗЕЛЁНЫМ. Пиннёный баг превращается в доказательство
фичи.

## 2. Зафиксированные дизайн-решения (Ф0)

1. **Гранулярность — точная: 1 байт поколения на 16-байтовую гранулу**
   (`idx = off / MIN_BLOCK`), 256 KiB на 4 MiB сегмент (~6–7% меты).
   Страничная гранулярность ОТВЕРГНУТА не по цене, а по корректности:
   соседи по странице бампали бы чужое поколение → легальные remote free
   дропались бы по false-mismatch → hardened ломал бы честные программы.
   Недопустимо: hardened может замедлять, но не может менять семантику
   легальных программ.
2. **Таблица живёт в метаданном регионе сегмента** (после alloc-bitmap,
   `#[cfg(feature = "hardened")]`), НЕ отдельным mmap:
   - не декоммитится с payload → поколения переживают decommit-reset
     (записки прошлой эры между reset и re-carve дропает существующий
     bump-guard; после re-carve нумерация продолжается — коллизий нет);
   - умирает только с полным release сегмента — а туда записки уже не
     доходят (`contains_base`/`magic_at` дропают раньше);
   - самохостинг-дисциплина сохранена, отдельного лайфцикла нет.
   Семантика ячеек: `AtomicU8`; owner пишет Relaxed (single-writer на
   выдаче), remote читает Relaxed. TSan-чисто, без барьеров.
3. **Бамп — в момент ВЫДАЧИ блока** (magazine pop + `pop_free`), не на
   free и не на carve: поколение = номер жизни. Carve не бампает — блок
   в магазине ещё «не выдан», его на этом этапе уже защищает X2-проверка
   членства в магазине. Free не бампает — иначе remote-read при легальном
   free читал бы уже-бампнутое значение.
4. **Раскладка записки (только под hardened):** off кратен MIN_BLOCK=16 →
   `off/16` = 18 бит; class ≤ 49 → 6 бит; **gen = 8 бит** (spare).
   `[gen:8 | class:6 | off16:18]` в том же u32 — ring не толстеет.
   Не-hardened раскладка не меняется вовсе.
5. **Wrap 1/256** — вероятностный residual (256 перерождений между
   запиской и дренажем → ложное совпадение). Фиксируется тестом-границей
   (стиль W7) и записывается в DURABILITY.md. Расширение gen (u64-записка)
   отвергнуто: ×2 к ring-футпринту ради и-так-UB-программ не окупается.

## 3. Фазы (коммит после каждой, zero-trust + судья)

### Ф0 — дизайн-док
Этот файл + при необходимости уточнения в §8.4 предшественника. Без кода.

### Ф1 — gen-таблица в метаданных
- Раскладка мета-региона: +256 KiB под `cfg(hardened)`; `small_meta_end`
  сдвигается; ёмкость сегмента под hardened падает (~7% меты) — тесты,
  зависящие от геометрии, аккуратно cfg-гейтить.
- Доступ: `gen_at(base, off)` / `bump_gen(base, off)` в стиле существующих
  field-specific аксессоров.
- Гейты: layout-тесты, miri (индексация/aliasing), **production-судья
  11/11 байт-в-байт** (весь код за флагом — чужие сборки не шелохнутся).

### Ф2 — перепаковка записки
- `pack_entry`/`unpack_entry` под hardened: `[gen:8|class:6|off16:18]`,
  const-asserts на разрядность (W7-стиль), round-trip юнит-тест.
- Не-hardened путь байт-в-байт нетронут (cfg-ветвление).

### Ф3 — СЕРДЦЕ: три касания + loom + counterfactual
- (а) выдача бампает: magazine pop и `pop_free` → `gen[off/16] += 1`;
- (б) remote free (`dealloc_routing` Variant-2) читает текущий gen
  (Relaxed) и вкладывает в записку;
- (в) дренаж (`reclaim_offset`/`_checked`) — ПОСЛЕ всех существующих
  гардов (magic/kind/align/bump/is_free/X2-магазин), ПЕРЕД `write_next`:
  `entry.gen != gen_at(off)` → drop (return false).
- loom: расширить композе-модель состоянием gen; инвариант «дренаж не
  линкует блок, чья текущая жизнь ≠ жизни записки (не-wrap)»;
  counterfactual-вариант без сверки обязан паниковать на A→B→I→D.
- Тесты: residual-тест A→B→I→D под hardened зеленеет (под остальными
  профилями остаётся `#[ignore]` — cfg-гейт); counterfactual в шапке.
- Эта фаза получает адверсариальный fl-аудит перед коммитом.

### Ф4 — швы жизненного цикла
- decommit-reset: непрерывность нумерации (тест: записка до reset,
  re-carve, дренаж — drop по gen, не по удаче);
- recycle/release: записки на умерший base дропаются существующими
  гардами (тест-подтверждение);
- adopt/abandon: таблица едет с сегментом (тест, если путь достижим).

### Ф5 — честные цены и границы
- Судья дважды: **production — байт-в-байт** (жёсткий гейт);
  **hardened — полная таблица цен в ledger** (бамп = RMW в холодную линию
  на каждом alloc; прогноз ±2–4% churn — измерить, не гадать; цена фичи
  публикуется, порога «не хуже» нет — это осознанная плата за защиту).
- Wrap-тест 1/256 (граница зафиксирована тестом).
- Доки: DURABILITY.md (+gen в инвентарь wrappable-счётчиков),
  FASTBIN_DESIGN/heap_core residual-ноты (сужение окна до мгновенной
  гонки вызова), CHANGELOG, дизайн-док §8.4 → «реализовано, см. этот план».
- Прогоны: TSan (remote-read vs owner-bump), miri, loom, полный suite
  под обоими профилями.

## 4. Риски и открытые вопросы

- **Ёмкость сегмента под hardened** меняет геометрию multiseg-бенча и
  любых тестов, считающих блоки-на-сегмент → инвентаризовать на Ф1.
- **Магазинный pop — самый горячий путь процесса**; бамп там = +1 RMW в
  таблицу (другая кэш-линия). Если hardened-цена окажется >5% churn —
  рассмотреть бамп на ВЫХОДЕ из магазина только под xthread-профилем
  (без alloc-xthread ринга нет — бамп не нужен вовсе; уточнить cfg:
  `all(hardened, alloc-xthread)`).
- **`pop_free` путь без магазина** (не-fastbin + hardened): бамп нужен и
  там — перечислить ВСЕ точки выдачи на Ф3 (как X2 перечислял дренажи;
  урок: пропущенный путь = дыра, ревью обязано перечислить).
- Wrap-и-совпадение 1/256 — принятая граница; документировать, не чинить.

## 5. Критерии готовности арки

1. Residual-тест зелен под hardened, `#[ignore]` под прочими.
2. Production-судья 11/11 байт-в-байт на каждой фазе.
3. Hardened-таблица цен опубликована в IAI_BASELINE-ledger.
4. loom green + counterfactual; TSan/miri чисты.
5. Доки синхронизированы; границы (мгновенная гонка, wrap 1/256)
   закреплены и тестом, и словами.
