## Находки (все лично верифицированы оркестратором по коду)

1. **P1 — R14-7's MAX_SEGMENTS raise (1024→4096) неполно измерен.**
   `WORDS_PER_CLASS`/`DIRTY_BITMAP_WORDS` выводятся из `MAX_SEGMENTS/64`
   (подтверждено: `segment_directory.rs:172`, `heap_slot.rs:350`) — оба
   выросли ×4 (16→64 слов). Footprint sidecar'ов: `PerClassDirty` ~6.3
   КиБ→~25 КиБ, `SegmentDirectory` под `numa-aware` ~55 КиБ→~220 КиБ на
   heap. `drain_dirty_segments` безусловно свипает scan_source целиком —
   ×4 атомарных RMW на каждый drain-проход. Стековый буфер в `Drop for
   AllocCore` реально стал 64 КиБ (было 16 КиБ) — комментарий на
   `alloc_core.rs:2101` до сих пор говорит «1024 × 16 B = 16 KiB stack»
   (подтверждено stale). Ни `IAI_BASELINE.md`, ни README bench:table не
   перегнаны после raise (единственное безусловное production-изменение
   раунда).
2. **P1 — `sidecar::reserve_zeroed_with` — safe fn с prose-only validity
   контрактом**, ровно тот класс дыры, ради устранения которого R14-9
   строился (подтверждено: `sidecar.rs`, `pub(crate) fn
   reserve_zeroed_with<T>(fixup: impl FnOnce(&mut T))` — НЕ `unsafe fn`,
   материализует `&mut T` над OS-нулевыми байтами ДО fixup). Единственный
   текущий вызов (`SegmentDirectory`, все поля integer/atomic) корректен,
   но примитив generic и `pub(crate)` — для будущего `T` с невалидным
   all-zero состоянием это once-call UB из safe-кода.
3. **P2 — `medium-classes` (Stage 2 promotion) × `exact-span-large` без
   headroom: дважды подряд «чинили тест», не код.** И
   `r14_4_promotion_move_leg_reduction.rs` (hotfix #302), и
   `r14_4_promotion_feature_off_non_disturbance.rs` (follow-up hotfix)
   упёрлись в один и тот же механизм: `exact-span-large` без
   `large-reserved-capacity` (или с `numa-aware`) даёt промотированному/
   растущему Large-блоку нулевой запас на рост — КАЖДЫЙ последующий grow
   релоцируется, даже мелкие шаги, которые без promotion остались бы
   in-place внутри medium-класса через OPT-F. Это потенциально строгая
   ПЕССИМИЗАЦИЯ этой комбинации фич, не просто «тест был неправ». Ответ
   уровня кода (cfg-решение или явный gate-замер) не сделан.
4. **P2 — новый loom-тест R14-2
   (`latch_trip_and_successful_publish_race_single_consumer_visit`) не
   доказывает заявленное.** Оба producer-потока `join()`-ятся ДО
   единственного визита консюмера — `join()` сам по себе даёт
   happens-before, поэтому assert «one-pass recovery» проходит при ЛЮБОМ
   ordering, включая полностью `Relaxed`. Тест не является контрпримером
   против ослабленного ordering — покрывает логику latch (обе записи
   восстанавливаются за один проход coarse-пути), но НЕ ordering-свойство,
   которое коммит-сообщение приписывает ему.
5. **P2 — пакет док-дрейфа после R14-7**, не ловится
   `no_stale_doc_references` (проверяет только README/ARCHITECTURE
   инвентарь, не произвольные числа в doc-комментариях):
   `segment_directory.rs:26,171` («WORDS_PER_CLASS=16», «MAX_SEGMENTS=1024»,
   «6,272 B / 55.1 KiB»), `dirty_by_class.rs:36-37` («16 ... 6.1 KiB»),
   `heap_slot.rs:209-210,344` («16 AtomicU64 words / 1024»),
   `alloc_core.rs:2101` («16 KiB stack» — реально 64 КиБ),
   `tests/regression_segment_table_tombstone_rebuild.rs:85-88`
   («HASH_CAPACITY=2048», «W=600 < MAX_SEGMENTS (1024)»).
6. **P3 (мелочи, объединить с #5 или сделать по возможности):**
   `sidecar::reserve`/`reserve_zeroed_with` не имеют compile-time
   `align_of::<T>() <= PAGE` (только проза); диапазон коммитов волны
   в задачах/докax снова указан с исключённой базой (тот же класс огреха,
   что fx-ревью отметило для Round 13) — гигиена на будущее, не
   исправлять задним числом.
7. **Процессная заметка (не находка кода):** к концу волны три висящих
   promotion-решения (R14-4 механизм — НЕ актуален, RED на kill-gate,
   promotion не предлагается; R14-5 CONDITIONAL-GO — тоже не предлагается
   по установленному правилу «только чистый GO спрашивается»; R14-6 —
   ЕДИНСТВЕННЫЙ чистый GO). Только R14-6 действительно требует
   AskUserQuestion — остальные два корректно не предлагались.
