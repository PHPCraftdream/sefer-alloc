## Саммари ревью №1 (внешнее, приоритезационное)

**Главный вывод:** Round 12 — «исправление + экспериментальная платформа», а не
радикальное общее ускорение production. В production реально вошли только
удаление обслуживания PageMap (R12-11: iai `multiseg_cold_256k` 490.1 → 329.2
Ir/op ≈ 1.49×; `seg_cycle_decommit_256k` 339.7 → 286.1 ≈ 1.19×; обычный churn
~1%) и primordial lazy commit (первый heap commit 4.535 → 0.883 MiB ≈ 5.14×,
только Windows). Радикальные результаты (exact-span 2–15.8×→~1.0× RSS,
class-aware-dirty 20–32×, reserved-capacity realloc) остаются opt-in.

**Находки (все подтверждены по коду):**

1. **P0 — OOM-переход class-aware-dirty теряет быстрый сигнал.** Producer при
   sidecar-OOM ставит только coarse-бит (`src/registry/heap_core_xthread.rs:369-380`),
   а consumer после успешной материализации sidecar другим producer'ом сканирует
   **исключительно** per-class slice (`src/alloc_core/alloc_core_small.rs:2437-2453`,
   `per_class_words.unwrap_or(ds)`). Запись из OOM-окна невидима быстрым путём до
   периодического полного скана (64 miss) / OOM-rescue — под давлением памяти до
   64 лишних сегментов (~256 MiB VA). Не UB (linear-scan backstop сохраняет
   корректность), но «OOM = только пропущенная оптимизация» — слишком сильное
   утверждение. → **#271 (R13-1)**.
2. **P1 — NUMA-карта: 8 узлов за всю жизнь, не одновременно.** `node_ids`
   append-only (`src/alloc_core/segment_directory.rs:280-289`); после 8
   когда-либо виденных узлов все последующие навсегда в unknown-bucket. R12-5
   (периодический refresh cached node) делает миграцию через 9+ узлов
   достижимее. Плюс стейл-комментарий «directory lookup под numa-aware
   ОТКЛЮЧЁН» (`src/alloc_core/alloc_core_small.rs:543-552`), противоречащий
   реализованному в R11-6 node-aware scan. → **#272 (R13-2)**.
3. **P1 — virgin-zero-skip обходит magazine для всех Small `alloc_zeroed`**
   (`src/registry/heap_core_alloc.rs:336-354`): на warm reuse memset всё равно
   выполняется И потерян tcache-hit — чистая регрессия steady-state; wall-clock
   gate не проводился. → **#273 (R13-3)**.
4. **P1 — вывод «page-run superseded» преждевременен**: exact-span-large и
   medium-classes-wide оба opt-in, в production 1.25–1.75 MiB по-прежнему идут
   через Large с 4 MiB-округлением. Корректный статус —
   deferred / no demonstrated production victim. → **#274 (R13-4)**.
5. **P2 — CI не изолирует новые feature-конфигурации** (только production и
   `--all-features`; комбинации маскируют друг друга — numa-aware исключает
   Windows reserved-capacity путь). → **#275 (R13-5)**.

**Направления дальнейшего ускорения:** production gate пары
exact-span + reserved-capacity (**#276**), расширение Large cache за пределы
8 слотов (**#277**), judge на 256–2048 живых объектов 260 KiB–2 MiB (**#278**),
promotion class-aware-dirty после фикса OOM-перехода (**#279**). Процессные:
production A/B/B/A отчёт после каждой волны, машинно-читаемые результаты,
гигиена комментариев (**#280**).

## Саммари ревью №2 (независимое, @fm, третий проход по той же волне)

Задача — новые находки поверх уже известных. Подтвердило одной строкой каждую
находку ревью №1 (и достаточность фикса R12-1: остальные держатели
`&SegmentDirectory` — диагностические). Отдельно проверило и **не нашло** проблем
в R12-4 (переполнений нет), R12-5, R12-6 (sweep идемпотентен), R12-9 (изоляция
Primordial от decommit-lifecycle реальна), virgin-предикате R12-10.

**Новые находки (все лично верифицированы оркестратором):**

- **N1 (P1, дефект ресурсов, не только перф):** virgin-ветка `alloc_zeroed`
  никогда не достигает `refill_magazine_slow`, а значит не вызывает
  `drain_heap_overflow()` (прелюдии живут в `heap_core_alloc.rs:439-449` для
  Small-magazine-miss и `:373-379` для Large-ветки). Heap, аллоцирующий только
  через `alloc_zeroed` (целевой calloc-профиль фичи!), никогда не дренирует свой
  HeapOverflow-ринг — cross-thread-freed блоки удерживаются неограниченно
  (реоткрывает M-9-класс ретеншена). → влито в **#273**, приоритет задачи
  повышен.
- **N2 (P2):** `clear_bit` (`segment_directory.rs:406`) использует
  **регистрирующий** `node_bucket_mut` вместо read-only `node_bucket` — no-op
  очистка от никогда не виденного узла зря съедает один из 8 регистрационных
  слотов, ускоряя исчерпание. Read-only lookup для clear корректен (у
  незарегистрированного узла битов нет). → влито в **#272**.
- **N3 (P2):** новый `tests/loom_class_aware_dirty.rs` (R12-7) **не вписан** в
  loom-job CI (`.github/workflows/ci.yml:515-534` перечисляет 6 файлов явно) и
  ни один loom-job не включает фичу `class-aware-dirty` — «loom-проверено»
  верно только как разовый локальный прогон. Уже второй случай (после #204)
  забытого loom-файла → нужна тест-инвентаризация tests/loom_*.rs ↔ ci.yml.
  → влито в **#275**, приоритет повышен до P1.
- **N4 (замечание):** loom_class_aware_dirty — ре-модель протокола на
  собственных атомиках, не реальные типы; зафиксировать в module-doc. → в #275.
- **Стейл README bench-таблица:** `npm run bench:table` последний раз перегнана
  2026-07-20 и помечена «post-Round9», а состав production с тех пор менялся
  дважды (R12-11 PageMap out, R12-9 primordial-lazy-commit in) — публичные числа
  README относятся к другой конфигурации. → влито в **#280** (плюс правило
  definition-of-done: волна, меняющая production-состав, перегоняет таблицу).
- **Large cache × exact-span:** 8-слотовый кэш с size-factor matching спроектирован
  под 4-MiB-квантование; под exact-span разброс размеров растёт и hit-rate
  упадёт — A/B гейт #276 может показать ложный проигрыш из-за кэша, а не самой
  оптимизации. → предостережение в **#276**, подтверждает приоритет **#277/#278**.
- Мелкое: линейный поиск по 8 слотам в `node_bucket_mut` на каждом
  set/clear_bit — не измерен; дешёвый last-node кэш попутно в #272.

полноценный результат задачи.
