## Саммари ревью №2 (независимое, внешнее, расширенное)

**Главный вывод:** ускорили не «весь аллокатор в 20×» — в production вошёл
один настоящий perf-механизм (class-aware-dirty), радикальный на узком
fan-in сценарии; обычный локальный churn / Large / single-thread этой волной
почти не изменены. 21.71× — throughput конкретного closed-loop judge, не
универсальная latency.

**Находки (все лично верифицированы):**

1. **P0 — небезопасная инициализация Large-cache extension** (совпадает с
   P1-1 ревью №1): `src/alloc_core/large_cache_extended.rs:140-142` (каст
   без `ptr::write`), `:108` (тип слотов). Дополнительно: `:152`/`:172` —
   safe-функции с unsafe caller contract, возвращающие `&'static`/
   `&'static mut` (лёгкий путь к пересекающимся ссылкам); фактическая
   ошибка модульной доки «AllocCore is Send» (сам тип намеренно
   `!Send + !Sync`, `src/alloc_core/alloc_core.rs:2102`). Фича opt-in —
   production не затронут, но это memory-safety blocker для неё.
2. **P0/P1 — loom доказывает не тот ordering, что в production**:
   producer `store(true, Release)` (`heap_core_xthread.rs:404`), док поля
   обещает пару Release/Acquire (`heap_slot.rs:294`), production-consumer
   читает `Relaxed` (`alloc_core_small.rs:2485`, со своим inline-аргументом
   достаточности), loom-модель читает `Acquire`
   (`tests/loom_class_aware_dirty.rs:586`). Не доказанный UAF (coarse
   bitmap + rescue консервативны), но заявленная гарантия видимости
   формально не закрыта, и три источника противоречат друг другу.
3. **P1 — 21.71× реально, но формулировка сильнее измерения**: judge —
   closed-loop, число owner allocations не фиксировано; criterion
   total-round интервалы при N=8 перекрываются (baseline 18.3–24.8 ms vs
   treatment 16.0–22.4 ms). Честная формулировка: «до ~20× owner allocation
   throughput на adversarial multi-class remote-free fan-in». Для будущих
   headline-утверждений нужен fixed-work process-level A/B/B/A judge.
4. **P1 — Large-cache extension может удерживать слишком много памяти**:
   `budget_bytes: None` по умолчанию (`large_cache_config.rs:147`), decay —
   не жёсткий предел; расширение 8→40 слотов может ×5 увеличить retained
   commit. Также нет гейта N=1/2/4 hit-path ПОСЛЕ материализации sidecar.
5. **P2 — stale NUMA-документация вводит CI в заблуждение**: ci.yml:209 и
   bench r12_7:64 утверждают «class-aware drain compiled out under NUMA»,
   но `drain_dirty_segments` не имеет `not(numa-aware)` гейта (проверено:
   гейт только `alloc-xthread` + ранний return при нулевом
   `directory_sidecar`); `--all-features` пропускает runtime-judge по
   неточной причине.
6. **P2 — отчёт смешивает production и opt-in fixes** (совпадает с P2-2
   ревью №1) + `git diff --check` красный (trailing whitespace,
   `docs/checkpoints/r13-complete.md:45` — проверено).

**Направления ускорения (в порядке приоритета ревью):** Stage 2
Small→Large realloc promotion (устранить 2111× realloc cliff, разблокировать
medium 31×/211×); безопасный production gate расширенного Large cache;
adaptive reserved capacity для exact-span (убрать +102% Ir, сохранить
15.8×→1.06× RSS); расширяемый SegmentTable. **Отложено осознанно:** batch
API (нет реального потребителя), page-run (сначала medium promotion).
