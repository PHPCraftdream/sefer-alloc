# Read-only аудит качества и архитектуры

Дата: 2026-07-12  
Объект: workspace `sefer-alloc`  
Метод: статическое чтение исходников и конфигурации. Git, сборки, тесты, бенчмарки, скрипты и fuzzing не запускались.

## Итог

Проект содержит сильную документацию инвариантов и заметную работу по локализации `unsafe`, но сопровождение осложняют три системные проблемы:

1. часть raw-pointer API объявлена безопасной, хотя корректность зависит от невыполнимых для safe API предусловий;
2. внутренности и тестовые швы раскрыты как `pub` ради интеграционных тестов, из-за чего `#[doc(hidden)]` ошибочно используется как граница API;
3. в основной ветке остаются отвергнутые эксперименты, future-only заготовки и большой объём исторических комментариев, увеличивающие feature matrix и затрудняющие чтение актуального алгоритма.

Ниже находки отсортированы по приоритету.

## Находки

### 1. `AllocCore::dealloc` и `AllocCore::realloc` ошибочно являются safe API

- Приоритет: критический.
- Файл и строки: `src/alloc_core/alloc_core.rs:671-721`, `src/alloc_core/alloc_core.rs:1248-1327`; публичный re-export — `src/lib.rs:273-280`.
- Обоснование: методы принимают произвольный `*mut u8`, но безопасность реализации требует, чтобы указатель был началом живого блока этого экземпляра и соответствовал `Layout`. Safe Rust-код вправе передать внутрь зарегистрированного сегмента interior pointer либо неверный `Layout`; проверка `contains_base` подтвердит только сегмент, после чего код читает/меняет метаданные или копирует `old_layout.size()` байт. Комментарий прямо формулирует контракт «previously returned», который нельзя возлагать на вызывающего safe-функции. Это тот же класс API, для которого `GlobalAlloc::dealloc/realloc` и `Allocator::deallocate/grow` являются `unsafe`.
- Предлагаемое улучшение: сделать `dealloc` и `realloc` `unsafe fn` с явным `# Safety`; аналогично проверить все публичные прокси в `HeapCore`. Если нужен действительно safe substrate, возвращать непрозрачный allocation token/handle, связывающий pointer, layout и владельца, и принимать его при освобождении.
- Риск текущего состояния: safe-клиент может вызвать UB или повредить allocator без единого `unsafe` в своём коде.
- Риск изменения: исправление сигнатур — breaking API change; потребуется обновить внутренние вызовы и клиентов. Token-based API существенно шире по объёму.
- Уверенность: высокая.

### 2. Test-only raw-pointer швы публичны и также оформлены как safe

- Приоритет: критический.
- Файл и строки: `src/lib.rs:234-257`, `src/alloc_core/mod.rs:48-59`, `src/alloc_core/remote_free_ring.rs:453-510`, `src/alloc_core/remote_free_ring.rs:592-599`, `src/alloc_core/numa.rs:38-63`; unsafe-контракт делегата — `crates/numa/src/lib.rs:164-177`.
- Обоснование: `alloc_core`, `global` и `registry` объявлены `pub` ради тестов. `#[doc(hidden)]` скрывает документацию, но не ограничивает доступ и не снимает semver/безопасностные обязательства. Например, safe `RemoteFreeRing::over_test_buffer` и `init_test_buffer` принимают произвольный raw pointer, а последующие safe методы создают `&'static AtomicU32` и пишут по нему. Аналогично публичный safe `alloc_core::numa::bind_segment` вызывает `unsafe numa_shim::bind_range`, полагаясь на внутренний инвариант владения reservation, которого внешний safe-клиент не обязан соблюдать.
- Предлагаемое улучшение: перенести white-box проверки в `#[cfg(test)]` unit tests рядом с кодом; интеграционными тестами проверять только поддерживаемый API. Если отдельный test-support слой необходим, вынести его в непубликуемый workspace crate или явно gated feature и всё равно сделать pointer-конструкторы `unsafe`. Вернуть внутренним модулям и типам `pub(crate)`.
- Риск текущего состояния: UB достижимо из внешнего safe-кода; внутренняя структура фактически становится публичной и затрудняет рефакторинг.
- Риск изменения: перенос большого числа интеграционных тестов потребует механической работы; закрытие уже доступных путей формально breaking change.
- Уверенность: высокая.

### 3. Конфигурация экземпляра `SeferAlloc` фактически является конфигурацией «первого вызова на потоке»

- Приоритет: высокий.
- Файл и строки: `src/global/sefer_alloc.rs:165-224`, `src/global/sefer_alloc.rs:235-255`, `src/global/tls_heap.rs:394-418`.
- Обоснование: `LargeCacheConfig` хранится в каждом `SeferAlloc`, но heap находится в общем TLS. Конфигурация применяется только при `LOCAL == null`; все последующие вызовы на потоке возвращают уже связанный heap без учёта того, через какой экземпляр allocator пришёл вызов. При прямом использовании нескольких `SeferAlloc` результат зависит от порядка первого вызова. API при этом допускает экземпляры и отдельно документирует сценарий нескольких экземпляров для `stats()`.
- Предлагаемое улучшение: либо сделать `SeferAlloc` ZST с process-wide конфигурацией, устанавливаемой один раз до первого bind, либо явно запретить/не поддерживать несколько конфигураций и документировать first-bind semantics. Более чистая альтернатива для прямого использования — instance-owned registry/TLS key, но это крупная смена архитектуры.
- Риск текущего состояния: тихое игнорирование ожидаемой конфигурации, недетерминированное по порядку вызовов на каждом потоке.
- Риск изменения: глобальная once-конфигурация требует политики конфликта; instance-owned registry увеличит память и стоимость TLS.
- Уверенность: высокая.

### 4. В основной кодовой базе сохранён отвергнутый `alloc-runfreelist` эксперимент

- Приоритет: высокий.
- Файл и строки: `Cargo.toml:204-235`, `src/alloc_core/mod.rs:60-76`, `src/alloc_core/run_stack.rs:1-48`, `src/alloc_core/alloc_core_small.rs:990-1180`, `src/alloc_core/alloc_core_small.rs:1837-1961`, `src/alloc_core/segment_header.rs:1030-1093`.
- Обоснование: manifest фиксирует NO-GO и регрессию 23–31%, но feature, 342-строчный модуль, альтернативные flush/drain алгоритмы, layout-ветви и отдельные тесты оставлены «для возможного будущего перезапуска». Это не мёртвый код для компилятора, но мёртвый продуктовый вариант: он расширяет матрицу конфигураций и заставляет каждое изменение allocator core учитывать заведомо не рекомендуемую реализацию.
- Предлагаемое улучшение: удалить feature и wiring из основной ветки; сохранить эксперимент в истории/отдельной research-ветке или отдельном непубликуемом crate. Если поддержка всё же обязательна, обозначить срок и владельца, исключить feature из публичного manifest до появления нового GO-решения.
- Риск текущего состояния: расхождение редко используемой ветки, рост стоимости review и вероятность ошибок в production-коде из-за `cfg`-раздвоений.
- Риск изменения: потеряется возможность мгновенно включить старую реализацию; внешние экспериментальные пользователи feature получат breaking change.
- Уверенность: высокая.

### 5. Legacy concurrent tier остаётся в продуктовой feature graph и утяжеляет `pinning`

- Приоритет: высокий.
- Файл и строки: `Cargo.toml:64-88`, `src/concurrent/mod.rs:5-15`, `src/concurrent/mod.rs:17-42`, `src/concurrent/epoch_region.rs:118-136`, `src/concurrent/lock_free_region.rs:147-151`, `src/concurrent/sharded_region.rs:167-171`.
- Обоснование: три семейства region помечены deprecated/legacy и «no new development», но feature `experimental` продолжает тянуть `arc-swap` и `crossbeam-epoch`. `pinning` зависит от всего `experimental`, хотя `PinnedRunner` использует только `core_affinity` и `ShardedRegion`; это уже признано в manifest как accepted over-inclusion. В результате устаревший research API продолжает влиять на зависимости, unsafe-аудит и сопровождение актуального pinning API.
- Предлагаемое улучшение: разделить features минимум на `sharded-region`, `epoch-region`, `snapshot-region` и `pinning`; затем вынести deprecated семейства в отдельный legacy crate либо удалить в следующей major версии. `pinning` должен зависеть только от узкого shard API и `core_affinity`.
- Риск текущего состояния: лишние зависимости и поверхность аудита, сложная feature matrix, невозможность независимо развивать полезный pinning слой.
- Риск изменения: feature split и удаление deprecated exports требуют миграционного периода и проверки совместимости комбинаций.
- Уверенность: высокая.

### 6. `LargeCacheMode` публикует два варианта, не меняющих поведение

- Приоритет: высокий.
- Файл и строки: `src/alloc_core/large_cache_mode.rs:11-31`, `src/alloc_core/large_cache_config.rs:238-251`, `src/alloc_core/alloc_core.rs:368-376`, `src/alloc_core/alloc_core_large_cache.rs:293-300`.
- Обоснование: `Background` и `Both` документированы как будущие режимы и сейчас идентичны `Lazy`. Значение только сохраняется и выдаётся диагностическим getter; ветвления поведения по mode нет. Публичный builder принимает настройку, которую реализация молча не исполняет.
- Предлагаемое улучшение: до реализации оставить только `Lazy` или заставить builder возвращать ошибку/отказ для неподдерживаемых режимов. Будущие варианты добавить вместе с поведением; `#[non_exhaustive]` уже позволяет расширить enum.
- Риск текущего состояния: конфигурационная ловушка — клиент считает, что background scavenger включён, хотя его нет.
- Риск изменения: удаление уже публичных variants — breaking change; мягкий путь — deprecated variants плюс явная runtime/compile-time диагностика.
- Уверенность: высокая.

### 7. `AllocBitmap` и `MagazineBitmap` почти полностью дублируют реализацию

- Приоритет: средний.
- Файл и строки: `src/alloc_core/alloc_bitmap.rs:44-138`, `src/alloc_core/magazine_bitmap.rs:58-143`.
- Обоснование: обе структуры имеют одинаковые поля, `FOOTPRINT`, `new`, byte-wise init, `locate`, test/set/clear и одинаковую арифметику адресации. Отличаются только доменные имена методов и комментарии. Исправление bounds/инициализации/геометрии легко применить лишь к одной копии.
- Предлагаемое улучшение: выделить приватный `SegmentBitmap` с `test/set/clear/init`; поверх него оставить тонкие semantic newtypes `AllocBitmap` и `MagazineBitmap`, чтобы не смешивать состояния на уровне типов.
- Риск текущего состояния: дрейф двух реализаций и удвоение review одинакового unsafe-membrane контракта.
- Риск изменения: чрезмерно общий тип может случайно смешать семантики; поэтому newtypes следует сохранить, а обобщить только механизм.
- Уверенность: высокая.

### 8. Документация `PageMap` противоречит фактической модели mixed-class pages

- Приоритет: средний.
- Файл и строки: `src/alloc_core/segment_header.rs:188-193`, `src/alloc_core/segment_header.rs:815-820`, `src/alloc_core/alloc_core_small.rs:39-47`, `src/alloc_core/alloc_core_small.rs:2119-2125`, `src/alloc_core/alloc_core_small.rs:2206-2224`.
- Обоснование: ранний комментарий утверждает, что страницы выделены одному классу, но carve-код реализует «first class wins», а актуальные комментарии прямо говорят, что общий bump cursor создаёт страницы с блоками нескольких классов и `PageMap` нельзя использовать для routing. Это не косметика: прежнее ошибочное понимание уже связано в комментариях с повреждением freelist.
- Предлагаемое улучшение: переписать описание `PageClass`/`PageMap` как non-authoritative first-touch hint, перечислить допустимые применения и запретить routing. По возможности переименовать `class_of` в `first_class_hint` либо типизировать назначение, чтобы старый инвариант нельзя было случайно восстановить по названию.
- Риск текущего состояния: будущий разработчик снова использует `PageMap` как источник истины и возвращает уже известный класс corruption bug.
- Риск изменения: переименование затронет тестовые швы и внутренние вызовы, но поведение менять не требуется.
- Уверенность: высокая.

### 9. Несколько module docs устарели и описывают уже несуществующую фазу реализации

- Приоритет: средний.
- Файл и строки: `src/alloc_core/run_stack.rs:5-21` против `src/alloc_core/alloc_core_small.rs:990-1180` и `src/alloc_core/alloc_core_small.rs:1837-1875`; `src/alloc_core/magazine_bitmap.rs:1-4` против `src/registry/heap_core.rs:790-791`, `src/registry/heap_core.rs:1360-1405`, `src/registry/heap_core.rs:2039-2047`.
- Обоснование: `run_stack.rs` утверждает, что flush/drain/reclaim ещё не используют `RunStack`, хотя Φ2/Φ3 wiring присутствует. `magazine_bitmap.rs` начинается условной фразой «не подключён, если NO-GO», хотя bitmap активно используется в production-gated `HeapCore`. Такие «фазовые снимки» быстро становятся ложными и заставляют читателя сверять исходники с внешним ledger.
- Предлагаемое улучшение: module docs должны описывать текущее состояние и стабильные инварианты. Историю фаз, task IDs и вердикты перенести в design/perf документы, оставив короткую ссылку и актуальный status (`active`, `experimental`, `rejected`).
- Риск текущего состояния: неверные решения при рефакторинге и недоверие к документации как источнику архитектуры.
- Риск изменения: минимальный; важно не удалить load-bearing safety rationale вместе с историей.
- Уверенность: высокая.

### 10. В production-модулях накоплены future-only функции и действительно неиспользуемая классификация Huge

- Приоритет: средний.
- Файл и строки: `src/registry/heap_core.rs:622-638`, `src/registry/heap_core.rs:2200-2223`, `src/alloc_core/alloc_core.rs:1543-1552`, `src/alloc_core/alloc_core_small_pool.rs:577-590`, `src/alloc_core/size_classes.rs:108-118`, `src/alloc_core/size_classes.rs:190-197`.
- Обоснование: методы `register_segment_internal`, `set_small_current_internal`, `reset_stamp_cache`, full-reset `decommit_empty_segment` и `is_huge` удерживаются с `allow(dead_code)` для гипотетических будущих фаз. При этом `SegmentKind` имеет только общий `Large` для large/huge (`src/alloc_core/segment_header.rs:159-169`), то есть Huge-классификация нигде не влияет на политику. Это YAGNI-код, который всё равно надо поддерживать при изменении инвариантов.
- Предлагаемое улучшение: удалить неиспользуемые wrappers/hooks и Huge classifier; вернуть их вместе с вызывающей функциональностью. Если отдельные seam-функции нужны только тесту, перенести тест к приватной реализации.
- Риск текущего состояния: ложное ощущение реализованной adoption/huge policy и дополнительная поверхность изменений.
- Риск изменения: будущую функцию придётся восстановить из истории и заново проверить на актуальных инвариантах — что безопаснее, чем молча использовать устаревшую заготовку.
- Уверенность: высокая.

### 11. Ключевые модули остаются чрезмерно крупными, а актуальный код утопает в исторических объяснениях

- Приоритет: средний.
- Файл и строки: `src/alloc_core/alloc_core_small.rs:1-2519`, `src/registry/heap_core.rs:1-2382`, `src/alloc_core/alloc_core.rs:1-1730`, `src/alloc_core/segment_header.rs:1-1693`; показательный блок — `src/registry/heap_core.rs:272-352`, крупный algorithm-history блок — `src/alloc_core/alloc_core.rs:787-918`.
- Обоснование: файлы объединяют entry points, routing, cache policy, diagnostics, test hooks и подробные ретроспективы отдельных task/bug IDs. Комментарии полезны для аудита, но сотни строк истории внутри struct/algorithm body ухудшают локальную читаемость и скрывают текущие предусловия. Принцип «one-export-per-file» не обеспечил разделение ответственностей: один export всё ещё содержит несколько подсистем.
- Предлагаемое улучшение: делить по поведению, а не по числу exports: small carve/refill/reclaim, heap routing, tcache, ownership stamping, diagnostics. В коде оставить короткие инварианты и safety proofs; bug history и benchmark verdicts перенести в ADR/design docs с устойчивыми ссылками.
- Риск текущего состояния: высокая когнитивная нагрузка, конфликтные изменения в одних файлах и трудный review cfg-комбинаций.
- Риск изменения: механическое разбиение allocator code может повлиять на inlining и codegen; проводить малыми шагами, сохраняя visibility и атрибуты hot-path функций.
- Уверенность: высокая.

### 12. Публичная документация `Region` неверно обещает dense/always-compact storage

- Приоритет: средний.
- Файл и строки: `src/lib.rs:3-18`, `crates/region/src/lib.rs:1-13`, `crates/region/src/region.rs:5-11`, фактический тип — `crates/region/src/region.rs:37-39`.
- Обоснование: документация многократно называет backing store dense, cache-friendly и always-compact, но используется `slotmap::SlotMap`, а не `slotmap::DenseSlotMap`. `SlotMap` сохраняет стабильные слоты и может содержать вакансии; тезис об always-compact backing вводит в заблуждение и влияет на ожидания от iteration/locality.
- Предлагаемое улучшение: либо исправить обещания на точные свойства `SlotMap`, либо осознанно перейти на `DenseSlotMap` после отдельной проверки семантики и производительности. Не использовать «dense» как маркетинговое свойство без соответствующего типа.
- Риск текущего состояния: неверные performance/space ожидания клиентов и ошибочные архитектурные решения поверх API.
- Риск изменения: переход на `DenseSlotMap` может изменить порядок итерации, стоимость remove и benchmark profile; простая правка документации риска почти не несёт.
- Уверенность: высокая.

### 13. Метаданные `sefer-region` неверно классифицируют crate как `no-std::no-alloc`

- Приоритет: низкий.
- Файл и строки: `crates/region/Cargo.toml:12-20`, `crates/region/src/lib.rs:35-45`.
- Обоснование: category объявляет `no-std::no-alloc`, но crate явно требует `extern crate alloc`, а `SlotMap` является аллоцирующей структурой. Собственная документация корректно говорит `no_std + alloc`.
- Предлагаемое улучшение: убрать `no-std::no-alloc` и выбрать категорию, соответствующую `no_std` с allocator, либо оставить только `data-structures`/`memory-management`.
- Риск текущего состояния: неправильная discoverability и ложное ожидание работы без global allocator.
- Риск изменения: только изменение metadata, runtime-риска нет.
- Уверенность: высокая.

### 14. `Reservation::is_empty` — заведомо бессмысленная часть публичного API

- Приоритет: низкий.
- Файл и строки: `crates/vmem/src/lib.rs:115-121`, invariant конструктора — `crates/vmem/src/lib.rs:239-260`.
- Обоснование: метод документирован как всегда возвращающий `false`, потому что zero-size reservation отвергается. Он существует только «для lint-friendliness» и не сообщает клиенту никакого состояния.
- Предлагаемое улучшение: удалить в следующей breaking версии. Если lint требует парности `len/is_empty`, локально разрешить lint или переосмыслить тип так, чтобы empty действительно был допустимым состоянием.
- Риск текущего состояния: небольшой API-шум и ложное предположение, что empty reservation возможен.
- Риск изменения: формально breaking change для редких вызовов метода; можно сначала deprecated.
- Уверенность: высокая.

### 15. Root `Cargo.toml` выполняет роль changelog/design ledger и затрудняет чтение конфигурации

- Приоритет: низкий.
- Файл и строки: `Cargo.toml:57-240`, `Cargo.toml:280-319`, `Cargo.toml:321-502`.
- Обоснование: определения features/targets окружены сотнями строк фаз, task IDs, отвергнутых решений, benchmark-истории и инструкций запуска. - Предлагаемое улучшение: оставить у каждой feature/target 1–3 строки актуального контракта и зависимостей; историю и инструкции перенести в отдельный feature-matrix/ADR документ. Добавить компактную таблицу поддерживаемых комбинаций в docs вместо narrative ledger в TOML.
- Риск текущего состояния: сложнее заметить ошибку feature graph при review; комментарии быстрее расходятся с реализацией.
- Риск изменения: при агрессивном сокращении можно потерять важное объяснение небезопасной комбинации; load-bearing ограничения (`fastbin -> alloc-xthread`) сохранить рядом с определением.
