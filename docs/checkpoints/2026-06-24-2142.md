## Decisions
- Shard unit = THREAD (lazy TLS binding), NOT core — robust to migration; pinning
  is an orthogonal apex optimization (shard==core). (User's insight, my sharpening.)
- Corrected the "hash key → shard" framing: our Region mints handles (no key), so
  the shard = the writer thread; a key-hashed `ShardedMap<K,V>` would be a
  different product, not a transformation of Region.
- Remote-free is central (not frontier): `AtomicSlot::evict`'s swap-to-null is the
  linearization point, so any-thread evict is sound — I2/I3 hold immediately;
  only the free-list re-add is owner-deferred via a crossbeam queue.
- Phase 7 staged 7a (zero unsafe baseline) → 7b (loom-gated remote-free depth) →
  7c/7d (apex + byte), mirroring the 3a→3b-II discipline. 7a alone = ~90% of the
  win for bounded long-lived thread pools.
- Chose to shard-replicate the EXISTING single-threaded vessels unchanged over
  making any structure lock-free — the apex of speed coincides with the least new
  unsafe (zero in 7a; only re-audited existing modules in 7b/7d).
