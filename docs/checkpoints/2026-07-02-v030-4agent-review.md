## Отчёты агентов (ПОЛНЫЙ КОНСПЕКТ — источник живёт только в транскрипте)

### Агент 1 — hot-path корректность + unsafe (272k токенов)
Прочитал alloc_core.rs, node.rs, os.rs, segment_header.rs, segment_table.rs,
alloc_bitmap.rs, size_classes.rs, segment_layout.rs, large_cache_config.rs +
смежные seams.
ОБЯЗАНЫ:
- **F1 CRITICAL** align≥SEGMENT (→ задача #130). alloc_core.rs:1762-1766,1887,1905;
  os.rs:126-135; vmem/lib.rs:329-366. align==1<<22: ptr=base+4MiB за
  зарегистрированной базой → dealloc no-op → утечка ≥8MiB+слот → 1024→abort.
  align>1<<23: база выровнена только до 4MiB → невыровненный ptr ~50% → UB.
  Guard `if align>=SEGMENT {null}` — S.
- **F2 HIGH** large_cache usable_size пересчёт из нового header → метка
  сжимается, span нет → RSS-амплификация неограниченная + врёт budget D1 (→
  задача #134). alloc_core.rs:505-511,1944-1950,1788. Фикс: span_usable в
  header.
- **F3 HIGH** HeapCore/fastbin ломает M2: (a) dealloc_routing читает magic_at
  до contains_base (heap_core.rs:1029) → чтение размапленной памяти при
  double-free опустевшего сегмента; (b) tcache word1 write по невыровненному
  interior-ptr = UB; (c) кросс-классовый DF → один ptr выдан дважды (→ #135).
- **F4 HIGH латентный UB** tcache-guard read_usize неинициализированного word1
  на каждом small-free (heap_core.rs:875) — miri поймал бы (не гоняется на
  fastbin). Штамповать word1 на refill/carve.
МОЖНО: F5 over-budget span вымывает весь кэш (#134); F6 ложные 'static SAFETY
в node.rs (recycle/#125 освобождают mid-process — комменты устарели); F7
tombstones hash не чистятся→full-scan (#135); F8 realloc O(n) scan (#135); F9
seq u64 wrap (астроном.); F10 стейл Drop-коммент; F11 M2 гранулярность базы;
F12 decommit не чистит ring.
Наблюдения (КОРРЕКТНО, не баг): alloc_zeroed зануляет large-cache-hit+magazine;
переполнений арифметики нет (Layout валидирует); carve через границы страниц
безопасен (page_map мёртвый груз); #126 index-scan безопасен; magazine↔decommit
закрыт (live_count учитывает magazine); M6 stale-guards полны; Drop без
double-release; A1 guard когерентен через reuse.
Топ-3: F1, F2, F3.

### Агент 2 — concurrency + memory ordering (213k токенов)
Прочитал heap_core.rs, heap_registry.rs, bootstrap.rs, heap_slot.rs,
tagged_ptr.rs, tcache.rs, tls_heap.rs, fallback.rs, remote_free_ring.rs,
segment_header.rs. Запустил loom (fallback_init 3/3, remote_ring 4/4).
ОБЯЗАНЫ:
- **#1 HIGH** stale LOCAL после AbandonGuard::drop → два писателя (→ задача
  #129). tls_heap.rs:88-97,119-160,215-224; heap_slot.rs:8-11,68-80.
  Generation-guard ЗАЯВЛЕН в доках heap_slot.rs но НЕ РЕАЛИЗОВАН (gen пишется
  claim:81, не читается на alloc). Err→Fallback МЁРТВ (Cell без Drop, try_with
  всегда Ok). Интерливинг: thread_local с Drop до первой аллокации → dtor
  после GUARD (LIFO) → recycle → stale free + конкурентный claim = 2×&mut.
  Фикс: TORN-сентинел в AbandonGuard::drop + маппинг→Fallback.
- **#2 MED-HIGH** ensure_slow OOM panic без rollback → вечный SENTINEL-спин;
  паника победителя аллоцирует → self-deadlock (→ задача #131).
  bootstrap.rs:298-299. Зеркало fallback F1-фикса.
A1 double-push guard РАЗБОР (задание §2): pre-reuse окно ГЕРМЕТИЧНО (2 ремня:
guard-CAS + magic==0 в кэше). Post-reuse double-free — реальный UAF, но класс
MED: унаследованный (own-thread имел всегда; small тоже не защищён post-reuse;
фундаментально нерешаем по адресу — free#2 старого == free#1 нового по адресу).
Guard 0.3.0 закрывает именно pre-drain self-loop→double-unmap (внутр.
корректность). Митигация: layout-vs-header (→ #138).
МОЖНО: #3 finish_bind провал GUARD.try_with→слот LIVE навсегда (лик, → #129);
#4 claim gen fetch_add до new → осиротевший слот с сожжённым gen; #5 mixed
atomic/non-atomic header в DF-окнах; #6 fallback спинлок паника в f→вечный
лок + нет backoff; #7 A1 drain только на Large-запросах (small-heap не
реклеймит); #8 wrap-теги (астроном.).
Наблюдения ordering: (а) A1 release-sequence цепочка корректна но по
нетривиальной причине (все head-мутации RMW) — записать комментом; (б) stamp
vs remote read owner_thread_free — HB транзитивно через app-передачу
указателя (задокументировать контракт); loom-модели ДРЕЙФУЮТ: A1 не
смоделирован, loom_thread_free = УДАЛЁННЫЙ Phase-10, loom_registry = мёртвый
adoption; tagged_ptr обещает ABA-модель — не написана (→ #138).
Топ-3: #1 stale-TLS, #2 ensure_slow, #3 A1 post-reuse (митигация+loom).

### Агент 3 — перф/ускорение (272k токенов)
Baseline замеры (production, 1024 пар): churn 16B Sefer 38.4µs vs mimalloc
56.5 (1.5×); 256B паритет; 1024B 34.8 vs 207.7 (6×); bulk 16B 32.2 vs 13.1
(2.5× медленнее — документ. worst-case магазина). Машина шумная ±15-20%.
ОБЯЗАНЫ-перф:
- **M1** глобальные fetch_add DBG_TCACHE_HITS (heap_core.rs:692) +
  LARGE_CACHE_HITS (alloc_core.rs:1799) на hit-путях = lock xadd на горячем
  пути (наш регресс C1/D1/E) → 20-30% ST + cross-core пингпонг MT бьёт по
  large_cache силе shamir-db (→ задача #133). Фикс: per-heap u64.
- **M2** HeapCore::realloc O(count) scan (heap_core.rs:985) при живом
  contains_base (→ #135).
- **M3** RemoteFreeRing::drain безусловный head.store даже когда пусто
  (remote_free_ring.rs:413) → когерентный трафик O(segments) на miss.
БЫСТРЫЕ ПОБЕДЫ: Q1 SegmentTable 3× O(count) scan на large-цикл→O(1) через
segment_id (#135); Q2 тройной class_for на alloc; Q3 runtime div в align_up
large-путь (align pow2→маска); Q4 Instant::now остаток при used>headroom
(штатно!) → сабсэмплинг.
LAYOUT/cache: L1 bump (owner) на одной линии с magic/kind/owner_tf
(remote-read) → task#33 развёл чтения не линии → reorder header #[repr(C)]
линия0=immutable, линия1=mutable; L2 ring курсоры+слоты на одной линии (false
sharing Вьюкова)→padding; L3 thread_free без padding.
СТРАТЕГИЧ: S1 find_segment O(n)→per-class hint/очереди; S2 элизия header-read
на proven-own dealloc (IDEA-4 §7.4, 3-8%); S3 alloc_zeroed skip memset свежих
OS-страниц.
Параметры: REFILL_BYTE_BUDGET 64KiB (свипнуть), RING_CAP 256 (если overflow>0),
LARGE_CACHE_SLOTS 8 (в LargeCacheConfig), bulk-mode без выхода (эксперимент).
Топ-3: M1 (убрать fetch_add), Q1+M2 (O(1) table+realloc), L1+L2+M3 (cache
lines cross-thread).
