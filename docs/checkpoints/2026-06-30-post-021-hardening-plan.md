## Decisions

- **Sequential sub-agents, не parallel, для Фаз A-F.** Фазы делят файлы
  (heap_core.rs: A+C; alloc_core.rs: C+D) и проектная методология требует
  gate "тесты + личный zero-trust review + коммит" между фазами — параллельный
  запуск сломал бы этот gate и создал бы конфликты правок. Отвергнуто:
  параллельный batch агентов на все 6 фаз сразу.
- **Агенты НЕ коммитят/не пушат.** Стандартное правило пользователя
  ("агенты не должны вызывать модифицирующие гит-команды"). Весь git — через
  главную сессию после личной проверки.
- **dbg-счётчики (relaxed AtomicU64, паттерн `DECOMMIT_CALLS`) раньше
  публичного stats() API.** Фазы A/B/D тестируют инварианты (reclaim,
  overflow, cache-hit) через `pub` diagnostic-статики без обёртки — Фаза E
  позже оборачивает всё в публичный `SeferAlloc::stats()`. Так тесты ранних
  фаз не блокируются, пока API стабилизируется.
- **Приоритет A > B > C > D > E > F** — по риску: перманентные утечки (A1)
  и unsound feature-комбо (A2) первыми (та же abort-сигнатура что #114, но
  ещё не покрыта), затем завершение архитектурного паттерна #114 для больших
  align (B), затем перф (C — рычаг именно для профиля пользователя),
  затем тюнинг лимитов (D), затем наблюдаемость (E — без неё утечки A/D
  невидимы в проде), затем CI-гигиена (F — важно, но не блокирует
  корректность).
- **Каждый фикс обязан иметь регрессионный тест с counterfactual
  (по возможности).** Где чистый counterfactual недостижим (C3 8 KiB memset
  — чисто перф без функционального различия) — честно помечено как
  "counterfactual невозможен, полагаемся на функциональный тест + бенч".
