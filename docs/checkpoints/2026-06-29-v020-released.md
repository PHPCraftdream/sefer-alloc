## Decisions

- **`SeferMalloc` → `SeferAlloc` via 0.2.0 breaking release.** Did the
  rename now (1 hour after 0.1.0 publish, ~no users yet) rather than
  carry the inconsistency. Yanked 0.1.0 so new installs pick up 0.2.0.
  Rejected: keep `SeferMalloc` as backward-compat alias — defeats the
  purpose of a clean rename, and 0.1.0 had effectively zero users.
- **`LargeCacheConfig` on hot path: by reference, not by value.**
  After the user's `he_repl` perf benchmark caught a 30% regression,
  fixed by changing `current_for_alloc_with_config` to take
  `&LargeCacheConfig`. Hot path now loads an 8-byte pointer instead
  of materialising the 40-byte struct on every alloc/dealloc site.
  Rejected: keep by-value and hope LLVM DCE's it — clearly didn't.
- **`malloc-bench-rs` sub-crate keeps its name.** It is a generic
  `GlobalAlloc` benchmark harness independently published; the name
  correctly reflects its purpose (works against any malloc-style
  allocator including mimalloc / jemalloc / SeferAlloc). Renaming would
  require re-publishing under a new name with no semantic gain.
- **Historical / planning doc bodies preserved as-is** even after
  file renames (e.g. `ALLOC_PLAN.md` body still references
  `SeferMalloc` and "malloc face" because it documents past phases
  when the type was named that way). Cross-references between docs
  updated; in-body type names in historical docs left alone. Rejected:
  blanket find/replace — would rewrite history.
- **macOS numa-shim CI fix via per-OS `MAP_ANON` cfg.** Bug
  pre-existed; CI was silently failing. Fixed properly (Linux 0x20,
  BSD family 0x1000) in `crates/vmem/src/lib.rs`. Rejected: skip the
  test on macOS — masks a real portability bug.

## Open questions

- **He_repl user-side wins**: ~22% of their runtime is SipHash on
  `HashSet<SmallVec<[u8;14]>>`. Suggested switching to FxHash; ball is
  in the user's court. Not a sefer-alloc concern.
- **Loom on the bootstrap CAS, weak-memory dimension.** The added
  `loom_bootstrap_cas` test models structural races (no-CAS double-
  alloc counterfactual) but loom 0.7 doesn't model weak-memory-model
  ordering violations (Relaxed vs Release). The Release/Acquire pair
  is verified transitively via TSan on `global_alloc_mt` and
  `heap_cross_thread`. Acceptable for 0.2.0; could re-evaluate
  if/when loom adds WMM coverage.
- **Trusted publishing via crates.io OIDC** — referenced as a
  future improvement in `release.yml` comments. Would remove the
  long-lived `CARGO_REGISTRY_TOKEN` secret; requires one-time UI
  setup per crate. Deferred.
- **`docs/ALLOC_PLAN.md` body still uses `SeferMalloc`**. As the
  canonical "main design doc" referenced from everywhere, it could
  arguably warrant a body rewrite to match current code. Left for
  future maintainer to decide.
